## Summary

Phase 3 completes the library's quality and release infrastructure. It requires three distinct workstreams: (1) Vitest unit tests with V8 coverage enforced to 100% on core files, (2) Playwright E2E tests across three real browsers using a static HTML fixture that loads the built `dist/`, and (3) GitHub Actions CI/CD with a pnpm-aware setup, Node matrix, and changesets release action.

The existing codebase is well-structured for testing. All five core files (`copy.ts`, `read.ts`, `detect.ts`, `fallback.ts`, `errors.ts`) follow a guard-first pattern with explicit branches, making 100% branch coverage achievable without heroics. The never-throw contract means every error path returns a typed value — test coverage maps directly to error codes.

The principal technical complexity is the E2E layer: clipboard operations require a real browser, a secure context (HTTPS or localhost), explicit permission grants (Chromium/WebKit), and careful Firefox handling (no clipboard permission API — `navigator.clipboard` works without explicit grant on localhost). The static HTML fixture approach (serving `dist/` via a local HTTP server embedded in Playwright's `webServer`) is simpler than a Vite dev server and avoids build-time coupling.

**Primary recommendation:** Build unit tests first to validate all branches, then layer on E2E to confirm real-browser behaviour, then wire CI. The release workflow can be added independently since `changesets/action@v1` configuration is already present in `.changeset/config.json`.

### Pattern 1: Vitest Unit Test with vi.stubGlobal

**What:** Mock `navigator.clipboard` globally per test file, restore in `afterEach`. Test each guard branch individually by controlling what the mock returns.

**When to use:** All unit tests for functions that call `navigator.clipboard`, `isBrowser()`, or `isSecureContext()`.

```typescript
// Source: vitest.dev/guide/mocking + CONTEXT.md D-02
import { afterEach, beforeEach, describe, expect, it, vi } from 'vitest'
import { copyToClipboard } from '../../src/clipboard/copy.ts'

describe('copyToClipboard', () => {
  const mockWriteText = vi.fn()

  beforeEach(() => {
    // Simulate browser environment
    vi.stubGlobal('navigator', {
      clipboard: { writeText: mockWriteText },
    })
    vi.stubGlobal('window', {
      isSecureContext: true,
    })
    mockWriteText.mockResolvedValue(undefined)
  })

  afterEach(() => {
    vi.unstubAllGlobals()
    vi.clearAllMocks()
  })

  it('returns true on success', async () => {
    expect(await copyToClipboard('hello')).toBe(true)
  })

  it('returns false when isBrowser() is false', async () => {
    vi.stubGlobal('navigator', undefined)
    vi.stubGlobal('window', undefined)
    expect(await copyToClipboard('hello')).toBe(false)
  })

  it('returns false on insecure context', async () => {
    vi.stubGlobal('window', { isSecureContext: false })
    expect(await copyToClipboard('hello')).toBe(false)
  })

  it('returns false when clipboard API is absent', async () => {
    vi.stubGlobal('navigator', { clipboard: undefined })
    expect(await copyToClipboard('hello')).toBe(false)
  })

  it('returns false on permission denied (NotAllowedError)', async () => {
    mockWriteText.mockRejectedValue(
      Object.assign(new DOMException('', 'NotAllowedError')),
    )
    expect(await copyToClipboard('hello')).toBe(false)
  })

  it('returns false on unexpected write error', async () => {
    mockWriteText.mockRejectedValue(new Error('unexpected'))
    expect(await copyToClipboard('hello')).toBe(false)
  })

  it('calls onError with CLIPBOARD_PERMISSION_DENIED on NotAllowedError', async () => {
    const onError = vi.fn()
    mockWriteText.mockRejectedValue(
      Object.assign(new DOMException('', 'NotAllowedError')),
    )
    await copyToClipboard('hello', { onError })
    expect(onError).toHaveBeenCalledWith(
      expect.objectContaining({ code: 'CLIPBOARD_PERMISSION_DENIED' }),
    )
  })

  it('does not throw when onError callback itself throws', async () => {
    mockWriteText.mockRejectedValue(new Error('fail'))
    const onError = vi.fn().mockImplementation(() => { throw new Error('boom') })
    await expect(copyToClipboard('hello', { onError })).resolves.toBe(false)
  })
})
```

### Pattern 3: Playwright Config with Three Browser Projects

**What:** Three projects (chromium, firefox, webkit) with per-project permission grants and a shared `webServer` serving the built `dist/`.

**When to use:** `playwright.config.ts` — cross-browser clipboard E2E.

```typescript
// Source: playwright.dev/docs/test-configuration [CITED: playwright.dev]
import { defineConfig, devices } from '@playwright/test'

export default defineConfig({
  testDir: './tests/e2e',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 1 : 0,
  reporter: process.env.CI ? 'github' : 'list',
  webServer: {
    command: 'npx http-server ./tests/e2e/fixtures -p 8080 --silent',
    url: 'http://localhost:8080',
    reuseExistingServer: !process.env.CI,
  },
  use: {
    baseURL: 'http://localhost:8080',
    trace: 'on-first-retry',
  },
  projects: [
    {
      name: 'chromium',
      use: {
        ...devices['Desktop Chrome'],
        // Clipboard permissions work in Chromium via grantPermissions
        contextOptions: {
          permissions: ['clipboard-read', 'clipboard-write'],
        },
      },
    },
    {
      name: 'firefox',
      use: {
        ...devices['Desktop Firefox'],
        // Firefox: clipboard-read/write are not valid permission strings —
        // navigator.clipboard works on localhost without explicit grant
      },
    },
    {
      name: 'webkit',
      use: {
        ...devices['Desktop Safari'],
        contextOptions: {
          permissions: ['clipboard-read', 'clipboard-write'],
        },
      },
    },
  ],
})
```

> **Firefox clipboard permission caveat:** `clipboard-read` and `clipboard-write` are not valid Playwright permission strings for Firefox. Attempting to grant them throws `Unknown permission`. Firefox allows clipboard access on `localhost` without explicit permission in Playwright's controlled context. [VERIFIED: github.com/microsoft/playwright/issues/13037]

### Pattern 4: E2E Test Using page.evaluate() with Static HTML Fixture

**What:** Static HTML fixture loads the built dist ESM file via `<script type="module">`. The test calls `page.evaluate()` to invoke library functions inside the browser context.

**Why static HTML over Vite devserver:** Avoids an additional build tool dependency in tests, directly exercises the dist output (what consumers get), simpler CI setup.

```html
<!-- tests/e2e/fixtures/index.html -->
<!DOCTYPE html>
<html lang="en">
<head><meta charset="UTF-8"><title>E2E Fixture</title></head>
<body>
  <script type="module">
    import * as clipboard from '/dist/clipboard/index.mjs'
    // Expose on window so page.evaluate can call them
    window.__clipboard = clipboard
  </script>
</body>
</html>
```

```typescript
// tests/e2e/clipboard.spec.ts
// Source: playwright.dev/docs/writing-tests [CITED: playwright.dev]
import { expect, test } from '@playwright/test'

test.describe('copyToClipboard', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/')
    // Wait for the module to load
    await page.waitForFunction(() => typeof window.__clipboard !== 'undefined')
  })

  test('returns true on HTTPS/localhost', async ({ page }) => {
    const result = await page.evaluate(() =>
      window.__clipboard.copyToClipboard('hello e2e'),
    )
    expect(result).toBe(true)
  })

  test('clipboard contains expected text after copy', async ({ page, context }) => {
    await page.evaluate(() =>
      window.__clipboard.copyToClipboard('verify-content'),
    )
    // Read back via Playwright clipboard API
    const text = await page.evaluate(() =>
      navigator.clipboard.readText(),
    )
    expect(text).toBe('verify-content')
  })
})

test.describe('isClipboardSupported', () => {
  test('returns true on localhost (secure context)', async ({ page }) => {
    await page.goto('/')
    await page.waitForFunction(() => typeof window.__clipboard !== 'undefined')
    const result = await page.evaluate(() =>
      window.__clipboard.isClipboardSupported(),
    )
    expect(result).toBe(true)
  })
})
```

> **Secure context for E2E:** `localhost` qualifies as a secure context in all browsers — Clipboard API is available without HTTPS. The fixture server at `http://localhost:8080` satisfies the `isSecureContext()` guard. [CITED: w3c.github.io/webappsec-secure-contexts/#is-origin-trustworthy]

> **Insecure context test:** Use a second `webServer` entry or `page.route` to intercept and reroute. Simpler approach: `vi.stubGlobal` in unit tests already covers this path (100% unit coverage handles the guard). The E2E insecure context test from D-05 can be omitted if the path is already 100% covered by unit tests — or confirmed via a separate HTTP-served page with a second Playwright project. **What:** Matrix strategy with ubuntu-latest × Node [20, 22] for unit tests. E2E as a separate job on ubuntu-latest + Node 22 with Playwright cache.

name: CI

on:

### Pattern 6: GitHub Actions Release Workflow (release.yml)

**What:** On push to `main`, `changesets/action@v1` either opens a Version PR (if changesets are present) or publishes to npm (if the Version PR was just merged).

```yaml
# .github/workflows/release.yml
# Source: github.com/changesets/action [CITED: github.com/changesets/action]
name: Release

on:
  push:
    branches: [main]

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

permissions:
  contents: write
  pull-requests: write

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # needed for changesets to see all commits
      - uses: pnpm/action-setup@v4
        with:
          version: 10
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'
          registry-url: 'https://registry.npmjs.org'
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - name: Create Version PR or Publish
        id: changesets
        uses: changesets/action@v1
        with:
          publish: pnpm publish --no-git-checks
          commit: 'chore: version packages'
          title: 'chore: version packages'
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
```

> **`--no-git-checks` flag:** Required because `changesets/action` has already committed version bumps — `pnpm publish` would otherwise complain about uncommitted changes. [ASSUMED — standard pattern for changesets + pnpm]

> **`registry-url`:** Set on `actions/setup-node` so that `NODE_AUTH_TOKEN` is automatically written to `.npmrc` for the publish step. This is required for npm auth in Actions. [CITED: github.com/actions/setup-node advanced-usage]

> **`permissions: contents: write` + `pull-requests: write`:** The changesets action needs these permissions to push commits and open PRs. When using `GITHUB_TOKEN`, these must be explicitly declared if the repo's default token permissions are read-only. [CITED: github.com/changesets/action]

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Browser binary caching | Custom cache key logic | `actions/cache@v4` with Playwright version hash | Correct cache invalidation, atomic restore |
| Permission grants per browser | Custom permission injection | `contextOptions.permissions` in Playwright config | Playwright handles browser-specific permission APIs |
| Coverage threshold enforcement | Script to parse coverage JSON | `coverage.thresholds` in vitest.config.ts | Built-in CI-fail behaviour, glob-pattern per-file control |
| npm auth in CI | Manual `.npmrc` creation | `registry-url` on `actions/setup-node` | Auto-injects `NODE_AUTH_TOKEN` into `.npmrc` |
| Version bumping | Manual `package.json` edits | `changeset version` via changesets action | Correct CHANGELOG.md generation, semver tracking |
| pnpm install cache | Manual `~/.pnpm-store` caching | `cache: 'pnpm'` on `actions/setup-node@v4` | Native integration, correct cache key from lockfile |

**Key insight:** Every problem above has a ~5-line standard solution. Hand-rolling any of them introduces subtle failures (stale caches, wrong npm auth, missed version bump in CHANGELOG) that only surface in production.

## Common Pitfalls

### Pitfall 1: Clipboard Permissions Differ by Browser in Playwright
**What goes wrong:** `grantPermissions(['clipboard-read', 'clipboard-write'])` throws `Unknown permission: clipboard-read` in Firefox and may be silently ignored in older WebKit versions.
**Why it happens:** Each browser engine has its own permission model. Firefox uses a prompt-based system for clipboard that Playwright cannot replicate via the CDP permissions API.
**How to avoid:** Set permissions only in chromium project config. For Firefox, rely on localhost's automatic clipboard access. For WebKit, test-verify permissions work — if they throw, fall back to localhost-only.
**Warning signs:** CI error message `Unknown permission` in Firefox E2E job. [VERIFIED: github.com/microsoft/playwright/issues/13037]

### Pitfall 2: navigator.clipboard is undefined on HTTP in Chrome
**What goes wrong:** In `copyToClipboard`, the guard `typeof navigator.clipboard?.writeText !== 'function'` needs to handle `navigator.clipboard` being `undefined` entirely (Chrome sets it to `undefined`, not just missing `writeText`, when on HTTP).
**Why it happens:** Chrome strips `navigator.clipboard` from the global on non-secure origins since Chrome 86+.
**How to avoid:** The existing code already uses optional chaining (`navigator.clipboard?.writeText`). Unit tests must cover the case where `navigator.clipboard` is `undefined`. [VERIFIED: reading copy.ts]
**Warning signs:** Missing branch coverage on the `typeof navigator.clipboard?.writeText !== 'function'` guard.

### Pitfall 3: vi.unstubAllGlobals() vs vi.restoreAllMocks() Confusion
**What goes wrong:** `vi.restoreAllMocks()` restores spies/mocks but does NOT restore `vi.stubGlobal` values. Using it in `afterEach` leaves global stubs in place and causes test bleed between files.
**Why it happens:** Two different mock registries in Vitest: spy registry (restoreAllMocks) and global stub registry (unstubAllGlobals).
**How to avoid:** Always call `vi.unstubAllGlobals()` in `afterEach` for tests that use `vi.stubGlobal`. Call both if the test uses both. [CITED: vitest.dev/guide/mocking]
**Warning signs:** Tests pass in isolation but fail in CI when run together.

### Pitfall 4: Coverage Missing Branches in fallback.ts DOM Branches
**What goes wrong:** `copyToClipboardLegacy` has a `document.body` check and a `textarea.isConnected` guard in `finally`. These require mocking both `isBrowser()` and `document` globals.
**Why it happens:** JSDOM (Vitest's default environment) provides `document.body` automatically — so the `!document.body` guard never returns `false` without explicit stubbing.
**How to avoid:** Use `vi.stubGlobal('document', { body: null })` to test the `!document.body` branch. Use a custom mock that removes the textarea before `finally` runs to test the `!textarea.isConnected` guard. [ASSUMED — based on code analysis]
**Warning signs:** Coverage report shows uncovered line on `document.body` null check.

### Pitfall 5: changesets/action Needs fetch-depth: 0
**What goes wrong:** `changesets/action` fails to detect which packages changed because `actions/checkout@v4` defaults to `fetch-depth: 1` (shallow clone).
**Why it happens:** Changesets compares against the base branch to determine what changed. Shallow clones have no history to compare against.
**How to avoid:** Add `fetch-depth: 0` to the checkout step in `release.yml`. [CITED: github.com/changesets/action]
**Warning signs:** Changesets action opens a version PR with no version bumps, or fails with a git error about missing refs.

### Pitfall 6: pnpm install --frozen-lockfile Fails When lockfile is Stale
**What goes wrong:** If `pnpm-lock.yaml` is not committed after adding `@playwright/test`, CI fails with `ERR_PNPM_OUTDATED_LOCKFILE`.
**Why it happens:** `--frozen-lockfile` is the correct CI mode (prevents accidental lockfile mutation) but requires the lockfile to already match `package.json`.
**How to avoid:** Always run `pnpm install` locally after adding any devDependency, and commit both `package.json` and `pnpm-lock.yaml` together.
**Warning signs:** CI failure on `pnpm install --frozen-lockfile` step.

### Pitfall 7: JSDOM vs Browser Environment for env.ts Tests
**What goes wrong:** `isBrowser()` returns `true` in JSDOM (Vitest's default environment) because JSDOM provides `window` and `navigator`. This makes it impossible to test the `!isBrowser()` path without stubbing.
**Why it happens:** Vitest uses JSDOM by default, which simulates a browser.
**How to avoid:** Stub both `navigator` and `window` to `undefined` using `vi.stubGlobal` to simulate a Node.js/SSR environment. Note: `vi.stubGlobal('window', undefined)` may affect other globals — test isolation via `afterEach(() => vi.unstubAllGlobals())` is critical. [ASSUMED — based on code analysis]
**Warning signs:** `isBrowser()` always returns `true` in tests, blocking branch coverage.

## Open Questions

1. **Insecure context E2E test implementation (D-05)**
   - What we know: Unit tests cover `isSecureContext()` returning `false` with 100% branch coverage. The `INSECURE_CONTEXT` code path is fully exercised in unit tests.
   - What's unclear: D-05 explicitly requires an E2E test "confirming functions return `false` on insecure context (HTTP served page)." This requires a second server on HTTP or `page.route` interception to override `window.isSecureContext`.
   - Recommendation: Use a second `webServer` entry in `playwright.config.ts` serving on HTTP (port 8081) and add a dedicated Playwright project `{ name: 'chromium-insecure', use: { baseURL: 'http://localhost:8081' } }`. Alternatively, `page.evaluate(() => Object.defineProperty(window, 'isSecureContext', { value: false }))` before calling library functions — simpler, avoids second server, but tests the stub not the real path.

2. **E2E fixture loading dist/ vs src/**
   - What we know: `dist/clipboard/index.mjs` is the published artifact. Loading it in E2E tests exercises the actual output.
   - What's unclear: If `dist/` is not built before E2E runs in local development, the fixture will fail to load. The CI workflow handles this (build step before E2E). Local dev needs `pnpm build` first.
   - Recommendation: Add a `prebuild` note in the E2E README section, or configure Playwright `webServer.command` to run `pnpm build && npx http-server ./tests/e2e/fixtures`.

3. **`test:e2e` script in package.json**
   - What we know: The `test:e2e` script is referenced in the release skill (`pnpm test && pnpm test:e2e`) but is not yet in `package.json`.
   - What's unclear: Whether to run with `--reporter=github` in CI or let config detect via `process.env.CI`.
   - Recommendation: Add `"test:e2e": "playwright test"` to `package.json` scripts. CI detection via `process.env.CI` in `playwright.config.ts` handles reporter switching.
